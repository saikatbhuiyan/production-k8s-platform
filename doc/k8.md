# Kubernetes Learning Roadmap

Kubernetes can feel overwhelming at first because it has many concepts, objects, and terms. A structured learning path makes it easier to understand how each part fits into the larger system.

For interview preparation and real-world usage, it is useful to learn Kubernetes in the following order:

```text
Core Concepts
  ↓
Control Plane Objects
  ↓
Workloads
  ↓
Networking
  ↓
Storage
  ↓
Configuration
  ↓
Security
  ↓
Scaling
  ↓
Extensions and Tools
```

---

# Why Learn Kubernetes This Way?

Kubernetes terminology can be difficult to understand when learned randomly.

A structured reference helps during:

* technical interviews
* architecture discussions
* troubleshooting sessions
* certification preparation
* production system design

Understanding where each Kubernetes object fits makes the system much easier to reason about.

---

# Main Areas to Focus On

For most backend, DevOps, and system design interviews, the most important Kubernetes areas are:

* core concepts
* workload resources
* configuration management
* storage
* networking
* scaling
* security basics

These areas cover most concepts that appear in real-world systems and interviews.

---

# 1. Core Concepts

Core concepts are the foundation of Kubernetes.

Before learning advanced topics, understand these first.

## Cluster

A Kubernetes cluster is a group of machines used to run containerized applications.

A cluster contains:

* control plane nodes
* worker nodes

The control plane manages the cluster.

Worker nodes run application workloads.

---

## Node

A node is a machine in the Kubernetes cluster.

A node can be:

* physical server
* virtual machine
* cloud instance

Worker nodes run Pods.

---

## Pod

A Pod is the smallest deployable unit in Kubernetes.

A Pod usually contains one container, but it can contain multiple tightly coupled containers.

Pods share:

* network namespace
* storage volumes
* lifecycle

Important point:

```text
You do not usually run containers directly in Kubernetes.
You run Pods.
```

---

## Container

A container is the packaged application runtime.

Kubernetes does not replace containers.

It manages containers through Pods.

---

## Namespace

A Namespace is a logical partition inside a Kubernetes cluster.

Namespaces are useful for:

* separating environments
* organizing resources
* applying access control
* limiting resource usage

Examples:

```text
dev
staging
production
monitoring
```

---

# 2. Control Plane Components

The control plane manages the desired state of the Kubernetes cluster.

---

## API Server

The API Server is the main entry point to the Kubernetes cluster.

All commands go through the API Server.

Examples:

```bash
kubectl get pods
kubectl apply -f deployment.yaml
```

Responsibilities:

* receives API requests
* validates requests
* updates cluster state
* communicates with other control plane components

---

## etcd

`etcd` is the key-value database used by Kubernetes.

It stores cluster state, including:

* Pods
* Deployments
* Services
* Secrets
* ConfigMaps
* node information

If `etcd` is lost, the cluster state is lost.

---

## Scheduler

The Scheduler decides where Pods should run.

It considers:

* available CPU and memory
* node labels
* taints and tolerations
* affinity rules
* resource requests

---

## Controller Manager

The Controller Manager runs controllers that keep the cluster in the desired state.

Example:

If a Deployment says there should be 3 Pods, but only 2 are running, a controller creates another Pod.

---

# 3. Worker Node Components

Worker nodes run application workloads.

---

## kubelet

`kubelet` runs on every worker node.

It is responsible for:

* starting Pods
* monitoring containers
* reporting node status
* communicating with the API Server

---

## kube-proxy

`kube-proxy` handles network rules for Services.

It helps route traffic to the correct Pods.

---

## Container Runtime

The container runtime runs containers.

Examples:

* containerd
* CRI-O
* Docker Engine

Modern Kubernetes commonly uses `containerd`.

---

# 4. Workload Resources

Workload resources define how applications run in Kubernetes.

---

## Deployment

A Deployment manages stateless applications.

It provides:

* rolling updates
* rollback support
* replica management
* self-healing

Example use case:

```text
API service
web application
backend worker
```

---

## ReplicaSet

A ReplicaSet ensures a specific number of Pod replicas are running.

Usually, you do not create ReplicaSets directly.

Deployments create and manage ReplicaSets.

---

## StatefulSet

A StatefulSet manages stateful applications.

It provides:

* stable Pod names
* stable network identity
* stable storage

Example use cases:

* databases
* Kafka
* Elasticsearch
* Redis clusters

---

## DaemonSet

A DaemonSet ensures that a Pod runs on every node or selected nodes.

Example use cases:

* logging agents
* monitoring agents
* security agents
* node-level networking tools

---

## Job

A Job runs a task until it completes successfully.

Example use cases:

* database migration
* batch processing
* one-time scripts

---

## CronJob

A CronJob runs Jobs on a schedule.

Example use cases:

* nightly reports
* scheduled backups
* cleanup tasks

---

# 5. Networking

Networking is one of the most important Kubernetes areas.

---

## Service

A Service provides a stable network endpoint for Pods.

Pods are temporary and may be recreated with new IPs.

Services solve this by providing stable access.

Types:

* ClusterIP
* NodePort
* LoadBalancer
* ExternalName

---

## ClusterIP

ClusterIP exposes a Service only inside the cluster.

It is the default Service type.

Example use case:

```text
Backend service used only by other services
```

---

## NodePort

NodePort exposes a Service on a port of every node.

Example:

```text
NodeIP:NodePort
```

Useful for simple external access, but not commonly used directly in production.

---

## LoadBalancer

LoadBalancer exposes a Service externally using a cloud provider load balancer.

Common in cloud environments.

---

## Ingress

Ingress manages HTTP/HTTPS access to services inside the cluster.

It is commonly used for:

* routing traffic by domain
* routing traffic by path
* TLS termination

Example:

```text
api.example.com → API service
app.example.com → frontend service
```

---

## Ingress Controller

Ingress rules do nothing by themselves.

An Ingress Controller implements those rules.

Examples:

* NGINX Ingress Controller
* Traefik
* HAProxy
* AWS Load Balancer Controller

---

## DNS

Kubernetes provides internal DNS.

Services can communicate using names like:

```text
user-service.default.svc.cluster.local
```

Usually, services in the same namespace can use:

```text
user-service
```

---

# 6. Configuration

---

## ConfigMap

A ConfigMap stores non-sensitive configuration.

Examples:

* application settings
* feature flags
* environment names
* external service URLs

ConfigMaps can be used as:

* environment variables
* mounted files

---

## Secret

A Secret stores sensitive configuration.

Examples:

* passwords
* API keys
* tokens
* certificates

Secrets are base64-encoded by default, not automatically strongly encrypted unless encryption at rest is configured.

---

## Environment Variables

Pods can receive configuration through environment variables from:

* ConfigMaps
* Secrets
* literal values

---

# 7. Storage

---

## Volume

A Volume provides storage to a Pod.

Unlike a container filesystem, a Volume can survive container restarts inside the same Pod.

---

## PersistentVolume

A PersistentVolume represents actual storage in the cluster.

It may come from:

* cloud disk
* NFS
* local storage
* storage provider

---

## PersistentVolumeClaim

A PersistentVolumeClaim is a request for storage by a workload.

The Pod uses the claim instead of directly selecting storage.

---

## StorageClass

A StorageClass defines how storage should be dynamically provisioned.

Example:

```text
fast-ssd
standard-disk
encrypted-storage
```

---

# 8. Security

---

## ServiceAccount

A ServiceAccount provides an identity for Pods.

Pods use ServiceAccounts to access the Kubernetes API.

---

## RBAC

RBAC means Role-Based Access Control.

It defines:

* who can access what
* which actions are allowed
* in which namespace or cluster scope

Key objects:

* Role
* ClusterRole
* RoleBinding
* ClusterRoleBinding

---

## NetworkPolicy

A NetworkPolicy controls traffic between Pods.

It can restrict:

* ingress traffic
* egress traffic

Useful for zero-trust networking inside the cluster.

---

## SecurityContext

SecurityContext defines security settings for Pods or containers.

Examples:

* run as non-root user
* read-only root filesystem
* Linux capabilities
* privilege escalation control

---

# 9. Scaling

---

## Horizontal Pod Autoscaler

Horizontal Pod Autoscaler, or HPA, adjusts the number of Pod replicas based on metrics.

Common metrics:

* CPU usage
* memory usage
* custom metrics

---

## Vertical Pod Autoscaler

Vertical Pod Autoscaler adjusts CPU and memory requests/limits for Pods.

It is useful when workload resource needs are unclear.

---

## Cluster Autoscaler

Cluster Autoscaler adjusts the number of nodes in the cluster.

If Pods cannot be scheduled due to insufficient resources, it may add nodes.

---

# 10. Scheduling

---

## Requests and Limits

Requests define the minimum resources a Pod needs.

Limits define the maximum resources a container can use.

Example:

```yaml
resources:
  requests:
    cpu: "250m"
    memory: "256Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

---

## Labels

Labels are key-value pairs attached to Kubernetes objects.

Used for:

* selection
* grouping
* filtering

Example:

```yaml
labels:
  app: user-service
  environment: production
```

---

## Selectors

Selectors are used to find objects based on labels.

Services use selectors to route traffic to matching Pods.

---

## Taints and Tolerations

Taints prevent Pods from being scheduled on certain nodes.

Tolerations allow Pods to run on tainted nodes.

Useful for:

* dedicated nodes
* GPU nodes
* special workloads

---

## Affinity and Anti-Affinity

Affinity controls where Pods prefer or require placement.

Anti-affinity prevents Pods from being placed together.

Useful for:

* high availability
* spreading replicas across nodes
* workload isolation

---

# 11. Health Checks

---

## Liveness Probe

Liveness probe checks if a container is still alive.

If it fails, Kubernetes restarts the container.

---

## Readiness Probe

Readiness probe checks if a container is ready to receive traffic.

If it fails, Kubernetes removes the Pod from Service endpoints.

---

## Startup Probe

Startup probe checks whether slow-starting applications have finished starting.

Useful for applications that take a long time to initialize.

---

# 12. Extensions and Tools

---

## Helm

Helm is a package manager for Kubernetes.

It helps install and manage applications using charts.

---

## Kustomize

Kustomize customizes Kubernetes YAML without templates.

Useful for managing environment-specific differences.

---

## Operators

Operators automate complex application management.

They are commonly used for:

* databases
* message brokers
* monitoring systems

---

## CRD

Custom Resource Definition allows users to create custom Kubernetes resource types.

Operators often use CRDs.

# Kubernetes Architecture

Kubernetes is organized as a **cluster**.

A Kubernetes cluster is composed of two main parts:

```text
Kubernetes Cluster
  ├── Control Plane
  └── Data Plane / Worker Nodes
```

The control plane manages the cluster.

The worker nodes run the actual application workloads.

---

# Control Plane vs Data Plane

## Control Plane

The control plane is the brain of Kubernetes.

It is responsible for:

* managing cluster state
* scheduling workloads
* handling API requests
* monitoring resources
* coordinating worker nodes
* ensuring the desired state matches the actual state

The control plane usually runs on one or more **master nodes**.

---

## Data Plane / Worker Nodes

The data plane is where application workloads run.

Worker nodes run:

* Pods
* containers
* application services
* Kubernetes node agents

In most clusters, there are usually more worker nodes than control plane nodes because application workloads are larger than Kubernetes system workloads.

---

# Master Nodes and Worker Nodes

## Master Node

A master node runs Kubernetes control plane components.

It contains the core Kubernetes system processes that manage the cluster.

Important control plane components include:

* API Server
* etcd
* Scheduler
* Controller Manager
* Cloud Controller Manager

For high availability, production clusters usually have multiple master nodes.

If there is only one master node and it fails, the Kubernetes control plane may become unavailable.

---

## Worker Node

A worker node runs application workloads.

Worker node components include:

* kubelet
* container runtime
* kube-proxy
* Pods
* Kubernetes workload objects

Worker nodes are where user applications actually run.

---

# Kubernetes Cluster Architecture

A simplified Kubernetes architecture looks like this:

```text
Developer / Operator
        |
        v
     kubectl
        |
        v
+--------------------------+
|      API Server          |
|   Kubernetes Control     |
|        Plane             |
+--------------------------+
        |
        | communicates with
        v
+--------------------------+
|      Worker Nodes        |
|                          |
|  kubelet                 |
|  kube-proxy              |
|  container runtime       |
|  Pods / Containers       |
+--------------------------+
```

# Control Plane Components

The control plane contains the components responsible for managing the cluster.

---

## API Server

The API Server is the main entry point into a Kubernetes cluster.

All communication with the cluster goes through the API Server.

Examples:

```bash
kubectl get pods
kubectl apply -f deployment.yaml
kubectl delete service api-service
```

These commands are translated into API requests and sent to the Kubernetes API Server.

The API Server is responsible for:

* receiving requests
* validating requests
* authenticating users
* authorizing actions
* updating cluster state
* communicating with etcd
* coordinating with other control plane components

---

## etcd

`etcd` is Kubernetes' distributed key-value store.

It stores the entire cluster state.

Examples of stored data:

* Pods
* Deployments
* Services
* ConfigMaps
* Secrets
* node information
* cluster configuration

Important point:

```text
If etcd data is lost, the Kubernetes cluster state is lost.
```

For production clusters, etcd should be backed up and run with high availability.

---

## Scheduler

The Scheduler decides where Pods should run.

When a new Pod is created, it may not yet be assigned to a worker node.

The Scheduler chooses the best node based on:

* available CPU
* available memory
* node labels
* taints and tolerations
* affinity and anti-affinity rules
* resource requests
* scheduling constraints

The Scheduler does not run the container directly.

It only decides which node should run the Pod.

---

## Controller Manager

The Controller Manager runs Kubernetes controllers.

Controllers continuously compare:

```text
desired state vs actual state
```

If the actual state does not match the desired state, controllers take action.

Example:

If a Deployment says:

```text
3 replicas should be running
```

but only 2 Pods are running, the controller creates another Pod.

The Controller Manager is responsible for keeping the cluster in the desired state.

---

## Cloud Controller Manager

The Cloud Controller Manager integrates Kubernetes with cloud providers.

It is responsible for cloud-specific operations such as:

* creating load balancers
* managing cloud volumes
* interacting with cloud nodes
* integrating with cloud networking

Examples:

* AWS
* Google Cloud
* Azure

This component is optional because not every Kubernetes cluster runs in a cloud environment.

---

# Worker Node Components

Worker nodes are responsible for running application workloads.

The main worker node components are:

* kubelet
* container runtime
* kube-proxy

These are mandatory components on worker nodes.

---

# kubelet

`kubelet` is an agent that runs on every worker node.

It communicates continuously with the control plane through the API Server.

The kubelet is responsible for:

* receiving Pod specifications
* starting containers inside Pods
* monitoring running containers
* reporting node and Pod status
* ensuring containers match the Pod specification

If Kubernetes says a Pod should run on a specific node, the kubelet on that node makes it happen.

---

## kubelet Example

If a Pod specification says:

```yaml
containers:
  - name: api
    image: my-api:1.0
```

then kubelet ensures that the container from image `my-api:1.0` is running inside the Pod.

---

# Container Runtime

The container runtime is responsible for actually running containers.

Kubernetes does not run containers directly.

It relies on a container runtime.

Common container runtimes include:

* containerd
* CRI-O
* Docker Engine

Modern Kubernetes commonly uses `containerd`.

The container runtime handles:

* pulling container images
* starting containers
* stopping containers
* managing container lifecycle

---

# kube-proxy

`kube-proxy` runs on each worker node.

It is responsible for maintaining network rules for Kubernetes Services.

kube-proxy helps route traffic from Services to the correct Pods.

It manages:

* service networking
* load balancing rules
* Pod traffic forwarding
* network connectivity inside the cluster

When a request reaches a Kubernetes Service, kube-proxy helps ensure that the request is forwarded to an appropriate Pod.

---

# Kubernetes Objects on Worker Nodes

Kubernetes also has many higher-level objects that define how workloads should behave.

Examples:

* Pods
* ReplicaSets
* Deployments
* Services
* Jobs
* StatefulSets
* DaemonSets
* Ingresses

These objects are optional in the sense that a cluster can technically exist without every one of them.

However, real-world Kubernetes applications usually use many of these objects.

---

# Pods

A Pod is the smallest deployable unit in Kubernetes.

A Pod can contain:

* one container
* multiple tightly coupled containers

Pods run on worker nodes.

Kubernetes does not usually run containers directly. It runs Pods, and Pods run containers.

---

# Deployment

A Deployment manages stateless applications.

It provides:

* replica management
* rolling updates
* rollback support
* self-healing

A Deployment usually creates and manages ReplicaSets, and ReplicaSets manage Pods.

---

# ReplicaSet

A ReplicaSet ensures that a specified number of Pod replicas are running.

Usually, ReplicaSets are managed by Deployments.

You normally do not create ReplicaSets directly.

---

# Service

A Service provides a stable network endpoint for accessing Pods.

Pods can be created, destroyed, and replaced.

Their IP addresses can change.

A Service provides a stable way to reach them.

---

# StatefulSet

A StatefulSet manages stateful applications.

It provides:

* stable Pod names
* stable network identity
* stable storage

Common use cases:

* databases
* Kafka
* Redis clusters
* Elasticsearch

---

# Job

A Job runs a task until it completes successfully.

Examples:

* database migration
* batch processing
* one-time scripts

---

# Ingress

Ingress manages HTTP and HTTPS access from outside the cluster to services inside the cluster.

It is commonly used for:

* domain-based routing
* path-based routing
* TLS termination

Example:

```text
api.example.com  -> API Service
app.example.com  -> Frontend Service
```

# How Kubernetes Components Communicate

The API Server is the central communication point in Kubernetes.

Most components communicate through the API Server.

```text
kubectl
  |
  v
API Server
  |
  |-- etcd
  |-- Scheduler
  |-- Controller Manager
  |-- Cloud Controller Manager
  |-- kubelet
  |-- kube-proxy
```

The API Server talks to:

* etcd
* Scheduler
* Controller Manager
* kubelet
* kube-proxy
* cloud integrations

Worker node components also report back to the API Server.

---

# Desired State vs Actual State

Kubernetes works by comparing desired state and actual state.

## Desired State

The desired state is what the user wants.

Example:

```yaml
replicas: 3
```

This means the user wants 3 Pods running.

---

## Actual State

The actual state is what is currently running in the cluster.

Example:

```text
Only 2 Pods are running
```

---

## Reconciliation

Kubernetes controllers continuously reconcile the difference.

If desired state is:

```text
3 Pods
```

and actual state is:

```text
2 Pods
```

then Kubernetes creates another Pod.

This reconciliation loop is one of the most important ideas in Kubernetes.

# kubectl

`kubectl` is the main command-line tool used to interact with a Kubernetes cluster.

It allows users to manage Kubernetes resources such as:

* Pods
* Services
* Deployments
* ConfigMaps
* Secrets
* Jobs
* StatefulSets
* Ingresses

`kubectl` works by sending API requests to the Kubernetes API Server.

---

# kubectl Command Structure

A typical `kubectl` command follows this structure:

```bash
kubectl <command> <resource-type> <resource-name> [flags]
```

Example:

```bash
kubectl get pods
```

Here:

| Part        | Meaning        |
| ----------- | -------------- |
| `kubectl` | Kubernetes CLI |
| `get`     | command        |
| `pods`    | resource type  |

Another example:

```bash
kubectl describe pod my-pod
```

Here:

| Part         | Meaning       |
| ------------ | ------------- |
| `describe` | command       |
| `pod`      | resource type |
| `my-pod`   | resource name |

---

# Main Types of kubectl Commands

kubectl commands can be grouped into three main categories:

1. Inspecting resources
2. Imperative resource management
3. Declarative resource management

---

# 1. Inspecting Resources

These commands are used to view the state of the cluster.

Common commands:

```bash
kubectl get
kubectl describe
kubectl logs
```

---

## kubectl get

Lists resources.

Examples:

```bash
kubectl get pods
kubectl get services
kubectl get deployments
kubectl get nodes
```

---

## kubectl describe

Shows detailed information about a resource.

Examples:

```bash
kubectl describe pod my-pod
kubectl describe deployment api-deployment
kubectl describe service api-service
```

Useful for debugging because it shows:

* events
* status
* labels
* selectors
* configuration
* errors

---

## kubectl logs

Shows logs from a Pod.

Examples:

```bash
kubectl logs my-pod
kubectl logs -f my-pod
```

For a specific container inside a multi-container Pod:

```bash
kubectl logs my-pod -c my-container
```

---

# 2. Imperative Resource Management

Imperative commands directly create, modify, or delete resources.

Examples:

```bash
kubectl run
kubectl create
kubectl scale
kubectl delete
```

---

## kubectl run

Creates and runs a Pod.

Example:

```bash
kubectl run nginx --image=nginx
```

---

## kubectl create

Creates a Kubernetes resource.

Example:

```bash
kubectl create deployment nginx --image=nginx
```

---

## kubectl scale

Scales a workload.

Example:

```bash
kubectl scale deployment nginx --replicas=3
```

---

## kubectl delete

Deletes a resource.

Example:

```bash
kubectl delete pod nginx
kubectl delete deployment nginx
```

---

# 3. Declarative Resource Management

Declarative management uses YAML manifests.

Instead of directly telling Kubernetes what command to run, you describe the desired state.

Then Kubernetes tries to make the actual state match it.

Main commands:

```bash
kubectl apply
kubectl diff
```

---

## kubectl apply

Applies a YAML manifest.

Example:

```bash
kubectl apply -f deployment.yaml
```

This creates or updates resources based on the file.

---

## kubectl diff

Shows what changes would be applied before actually applying them.

Example:

```bash
kubectl diff -f deployment.yaml
```

This is useful for reviewing changes before applying them.

---

# Imperative vs Declarative

## Imperative

You tell Kubernetes exactly what action to perform.

Example:

```bash
kubectl create deployment nginx --image=nginx
```

Good for:

* quick testing
* learning
* temporary resources

---

## Declarative

You define the desired final state in YAML.

Example:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3
```

Then apply:

```bash
kubectl apply -f deployment.yaml
```

Good for:

* production
* GitOps
* repeatable deployments
* version control

# Kubernetes Pods

Pods are the foundation of Kubernetes workloads.

A Pod is the smallest and simplest unit that can be created in Kubernetes to run containers.

In Kubernetes, containers do not run directly by themselves. They must run inside a Pod.

```text
Container
  ↓
Pod
  ↓
Kubernetes Worker Node
```

---

# What Is a Pod?

A Pod represents a single instance of a running process in a Kubernetes cluster.

A Pod can contain:

* one container
* multiple closely related containers

Most Pods run a single main application container.

However, a Pod can also include additional helper containers when those containers need to work closely with the main container.

---

# Why Pods Exist

Kubernetes uses Pods as an abstraction above containers.

Pods provide:

* shared networking
* shared storage
* lifecycle management
* container configuration
* health checks
* resource configuration
* security configuration

Instead of managing containers directly, Kubernetes manages Pods.

---

# Single-Container Pod

The most common Pod pattern is a single-container Pod.

Example:

```text
Pod
 └── Container: API Server
```

This is common for:

* backend APIs
* web servers
* workers
* frontend applications

---

# Multi-Container Pod

A Pod can contain multiple containers.

Example:

```text
Pod
 ├── Main Container: Application
 └── Sidecar Container: Logging Agent
```

Multiple containers should be placed in the same Pod only when they are tightly coupled and need to share resources.

Common examples:

* logging sidecar
* monitoring sidecar
* proxy sidecar
* data loader
* initialization helper

---

# When Not To Use Multiple Containers in One Pod

You should usually not run two independent applications inside the same Pod.

Bad example:

```text
Pod
 ├── User Service
 └── Payment Service
```

These should normally be separate Pods:

```text
User Service Pod
Payment Service Pod
```

Reason:

* each service should scale independently
* each service should deploy independently
* failures should be isolated
* ownership is cleaner

---

# Shared Network

Containers inside the same Pod share the same network namespace.

This means containers in the same Pod can communicate using:

```text
localhost
```

Example:

```text
Container A → localhost:3000 → Container B
```

This works because both containers share the Pod network.

---

# Port Usage Inside a Pod

Containers inside the same Pod share the same IP address.

Because of this, containers in the same Pod should not listen on the same port.

Example:

```text
Container A listens on port 3000
Container B listens on port 3001
```

This is valid.

But this is invalid:

```text
Container A listens on port 3000
Container B also listens on port 3000
```

Port conflicts should be avoided.

---

# Pod IP Address

Each Pod gets its own IP address inside the Kubernetes cluster.

By default:

* Pods can communicate with other Pods
* communication can happen across nodes
* communication can happen across namespaces

This default behavior can be restricted using NetworkPolicies.

---

# Shared Storage

Containers inside the same Pod can share volumes.

A volume can be mounted into one or more containers.

Example:

```text
Pod
 ├── Container A
 │    └── mounts /data
 └── Container B
      └── mounts /data
```

Both containers can read from the same volume.

Depending on configuration, they may also write to it.

---

# Read-Only and Read-Write Volume Mounts

Volume mounts can be configured as:

* read-only
* read-write

Example use cases:

## Read-only

A configuration file mounted into multiple containers.

## Read-write

A shared directory where one container writes data and another container reads it.

---

# Types of Storage in Pods

Pods can use different types of storage.

Common examples:

* ephemeral storage
* emptyDir volume
* ConfigMap volume
* Secret volume
* PersistentVolumeClaim

---

## Ephemeral Storage

Ephemeral storage exists only while the Pod exists.

If the Pod is deleted, the data is lost.

Useful for:

* temporary files
* cache files
* scratch space

---

## Persistent Storage

Persistent storage survives Pod restarts and replacements.

This is usually provided through:

```text
PersistentVolume
PersistentVolumeClaim
```

Useful for:

* databases
* uploaded files
* durable application data

---

# Pod Configuration

Pods allow configuration of how containers run.

Common configuration includes:

* container image
* ports
* environment variables
* volumes
* resource requests
* resource limits
* security settings
* health probes
* startup command
* arguments

---

# Environment Variables

Containers inside Pods can receive environment variables.

Example use cases:

* application environment
* service URLs
* feature flags
* runtime configuration

Example:

```yaml
env:
  - name: NODE_ENV
    value: production
```

---

# Container Ports

Pod specs can define which ports containers expose.

Example:

```yaml
ports:
  - containerPort: 3000
```

This documents which port the container listens on.

However, exposing a container port in a Pod does not automatically expose it outside the cluster.

For stable access, Kubernetes usually uses a Service.

---

# Resource Requests and Limits

Pods can define CPU and memory requests and limits.

## Requests

Requests define the minimum resources required for scheduling.

Example:

```yaml
resources:
  requests:
    cpu: "250m"
    memory: "256Mi"
```

Kubernetes uses requests to decide which node can run the Pod.

---

## Limits

Limits define the maximum resources a container can use.

Example:

```yaml
resources:
  limits:
    cpu: "500m"
    memory: "512Mi"
```

Limits prevent one container from consuming too many resources on a node.

---

# Health Probes

Kubernetes supports health checks for containers.

Common probe types:

* liveness probe
* readiness probe
* startup probe

---

## Liveness Probe

A liveness probe checks if the container is still alive.

If the liveness probe fails, Kubernetes restarts the container.

Useful when:

* application is stuck
* process is running but unhealthy
* service needs automatic recovery

---

## Readiness Probe

A readiness probe checks if the container is ready to receive traffic.

If the readiness probe fails, Kubernetes removes the Pod from Service endpoints.

Useful when:

* application is starting
* dependencies are not ready
* service should temporarily stop receiving traffic

---

## Startup Probe

A startup probe checks whether the application has finished starting.

Useful for slow-starting applications.

It prevents Kubernetes from killing the container too early before it finishes initialization.

---

# Pod Lifecycle

A Pod goes through different phases.

Common Pod phases:

| Phase     | Meaning                                     |
| --------- | ------------------------------------------- |
| Pending   | Pod accepted but not fully running          |
| Running   | Pod assigned to node and containers running |
| Succeeded | All containers completed successfully       |
| Failed    | At least one container failed               |
| Unknown   | Pod state cannot be determined              |

---

# Pods Are Usually Not Managed Directly

Although you can create Pods directly, in real-world Kubernetes usage, Pods are usually managed by higher-level resources.

Common controllers:

* Deployment
* ReplicaSet
* StatefulSet
* DaemonSet
* Job
* CronJob

Example:

```text
Deployment
  ↓
ReplicaSet
  ↓
Pods
  ↓
Containers
```

---

# Why Not Create Pods Directly?

Directly created Pods are not automatically replaced if they fail or are deleted.

For example:

```bash
kubectl run nginx --image=nginx
```

This creates a Pod.

But in production, you usually use a Deployment because it provides:

* self-healing
* replica management
* rolling updates
* rollback support

---

# Pod Example YAML

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx:latest
      ports:
        - containerPort: 80
```

This defines a Pod running one Nginx container.

---

# Multi-Container Pod Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-sidecar
spec:
  containers:
    - name: app
      image: my-app:1.0
      ports:
        - containerPort: 3000

    - name: log-sidecar
      image: logging-agent:1.0
```

This Pod has:

* one main application container
* one sidecar container

Both containers share the same Pod network and can share volumes if configured.

---

# Pod With Shared Volume Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: shared-volume-pod
spec:
  volumes:
    - name: shared-data
      emptyDir: {}

  containers:
    - name: writer
      image: busybox
      command: ["sh", "-c", "echo hello > /data/message && sleep 3600"]
      volumeMounts:
        - name: shared-data
          mountPath: /data

    - name: reader
      image: busybox
      command: ["sh", "-c", "cat /data/message && sleep 3600"]
      volumeMounts:
        - name: shared-data
          mountPath: /data
```

Both containers mount the same volume at `/data`.

---

# Communication Between Containers in Same Pod

Containers inside the same Pod can communicate using `localhost`.

Example:

```text
Main app container listens on port 3000
Sidecar container calls http://localhost:3000
```

This is possible because containers in the same Pod share the same network namespace.

---

# Communication Between Pods

Pods communicate using Pod IPs or Services.

However, Pod IPs are temporary.

Because Pods can be recreated with new IPs, applications should usually communicate through Services.

Example:

```text
frontend Pod → backend Service → backend Pods
```

---

# Important Characteristics of Pods

## Pods Are Ephemeral

Pods are not permanent.

They can be:

* deleted
* restarted
* rescheduled
* replaced

Applications should not assume a Pod will live forever.

---

## Pods Have Their Own IP

Each Pod gets a unique cluster IP.

Containers inside the Pod share that IP.

---

## Pods Can Be Replaced

If a Pod crashes, a controller such as Deployment may create a new Pod.

The new Pod may have:

* a different name
* a different IP
* a fresh filesystem

---

## Pods Are Scheduling Units

Kubernetes schedules Pods onto nodes.

The Scheduler does not schedule containers individually.

It schedules Pods.

---

# Pod vs Container

| Concept                         | Pod                      | Container           |
| ------------------------------- | ------------------------ | ------------------- |
| Kubernetes unit                 | Smallest deployable unit | Runtime process     |
| Can run directly in Kubernetes? | Yes                      | No                  |
| Contains                        | One or more containers   | Application process |
| Network                         | Has Pod IP               | Shares Pod network  |
| Storage                         | Defines volumes          | Mounts volumes      |
| Lifecycle                       | Managed by Kubernetes    | Managed inside Pod  |

---

# Pod Best Practices

## Use One Main Application Per Pod

A Pod should usually represent one application or one tightly coupled unit of work.

---

## Use Sidecars Only When Needed

Use sidecars for supporting responsibilities such as:

* logging
* metrics
* proxying
* service mesh
* data synchronization

Avoid using sidecars for unrelated applications.

---

## Use Deployments for Long-Running Apps

For APIs, web apps, and workers, use Deployments instead of standalone Pods.

---

## Configure Resource Requests and Limits

Always define CPU and memory requests and limits in production.

This helps Kubernetes schedule workloads correctly and prevents resource exhaustion.

---

## Configure Health Probes

Use readiness and liveness probes for production workloads.

This improves:

* self-healing
* safe traffic routing
* deployment stability

---

## Avoid Storing Important Data Inside Pod Filesystem

Pod filesystems are temporary.

Use persistent volumes for durable data.

---

# Summary

Pods are the smallest deployable units in Kubernetes.

Key points:

* Kubernetes runs containers through Pods.
* A Pod can contain one or more containers.
* Containers inside the same Pod share network and storage resources.
* Containers in the same Pod can communicate through `localhost`.
* Pods can define ports, environment variables, volumes, security settings, resources, and health probes.
* Pods are ephemeral and replaceable.
* In production, Pods are usually managed by Deployments, StatefulSets, DaemonSets, Jobs, or CronJobs.

A good mental model:

```text
Pod = wrapper around one or more containers
```

Kubernetes does not manage individual containers directly.

It manages Pods.


# Kubernetes Manifests / Configuration Files

Kubernetes objects can be created and managed in different ways.

One of the most important ways is by using **manifest files**, also called **configuration files**.

In Kubernetes, the terms **manifest file** and **configuration file** are often used interchangeably.

A manifest file defines the **desired state** of a Kubernetes object.

For example, a manifest can describe:

* a Pod
* a Service
* a Deployment
* a Namespace
* a ConfigMap
* a Secret
* an Ingress

Kubernetes then reads that file and tries to create or update the actual cluster state to match what is described in the manifest.

---

# Why Manifest Files Matter

Manifest files are the foundation for most Kubernetes work.

They allow us to define infrastructure and workloads as code.

Instead of manually creating objects one by one using commands, we can describe what we want in YAML files and apply them to the cluster.

This makes Kubernetes configuration:

* repeatable
* version-controlled
* reviewable
* easier to automate
* easier to share across environments

---

# Imperative vs Declarative Management

Kubernetes supports both **imperative** and **declarative** ways of managing resources.

---

## Imperative Management

In the imperative approach, we directly tell Kubernetes what action to perform.

Example:

```bash
kubectl run nginx-pod --image=nginx
```

This command directly tells Kubernetes:

```text
Create a Pod named nginx-pod using the nginx image.

Other imperative examples:

```bash
kubectl create deployment nginx --image=nginx
kubectl expose pod nginx-pod --type=NodePort --port=80
kubectl delete pod nginx-pod
kubectl scale deployment nginx --replicas=3
```

Imperative commands are useful for:

* quick testing
* learning Kubernetes
* debugging
* generating initial manifests

However, imperative commands are harder to track and reproduce over time.

---

## Declarative Management

In the declarative approach, we define the desired state in a YAML manifest file.

Example:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
spec:
  containers:
    - name: nginx
      image: nginx
```

Then we apply it:

```bash
kubectl apply -f nginx-pod.yaml
```

Here, we are not telling Kubernetes each step to execute.

Instead, we are saying:

```text
This is the desired state. Make the cluster match it.
```

Declarative management is preferred for:

* production environments
* GitOps workflows
* CI/CD pipelines
* repeatable deployments
* infrastructure as code

---

# What Is a Kubernetes Manifest?

A Kubernetes manifest is a YAML file that describes a Kubernetes object.

It tells Kubernetes:

* what type of object to create
* what name the object should have
* which namespace it belongs to
* what labels and annotations it should use
* what configuration the object should follow

Example:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx
      ports:
        - containerPort: 80
```

This manifest defines a Pod named `nginx-pod`.

---

# YAML Syntax

Kubernetes manifests are usually written in YAML.

YAML is indentation-sensitive.

Important rules:

* indentation matters
* use spaces, not tabs
* lists use `-`
* key-value pairs use `key: value`
* nested objects are indented

Example:

```yaml
metadata:
  name: nginx-pod
  labels:
    app: nginx
```

Here:

* `name` is inside `metadata`
* `labels` is inside `metadata`
* `app` is inside `labels`

---

# Main Top-Level Fields

Most Kubernetes manifests contain four main top-level fields:

```yaml
apiVersion:
kind:
metadata:
spec:
```

These fields are common across many Kubernetes objects.

However, not every Kubernetes object has all four fields.

For example, some objects like `Namespace` may not need a `spec` field.

---

# 1. apiVersion

The `apiVersion` field tells Kubernetes which API group and version should be used for the object.

Example:

```yaml
apiVersion: v1
```

or:

```yaml
apiVersion: apps/v1
```

The API version determines which version of the Kubernetes API will process the manifest.

---

## API Group and Version

The `apiVersion` can include:

```text
API group + API version
```

Example:

```yaml
apiVersion: apps/v1
```

Here:

* `apps` is the API group
* `v1` is the API version

For core Kubernetes objects, the API group is omitted.

Example:

```yaml
apiVersion: v1
```

This belongs to the core Kubernetes API group.

Common examples:

| Object      | apiVersion             |
| ----------- | ---------------------- |
| Pod         | `v1`                   |
| Service     | `v1`                   |
| ConfigMap   | `v1`                   |
| Secret      | `v1`                   |
| Namespace   | `v1`                   |
| Deployment  | `apps/v1`              |
| ReplicaSet  | `apps/v1`              |
| StatefulSet | `apps/v1`              |
| DaemonSet   | `apps/v1`              |
| Job         | `batch/v1`             |
| CronJob     | `batch/v1`             |
| Ingress     | `networking.k8s.io/v1` |

---

# 2. kind

The `kind` field defines what type of Kubernetes object is being created or managed.

Example:

```yaml
kind: Pod
```

or:

```yaml
kind: Deployment
```

The `kind` must be supported by the specified `apiVersion`.

For example:

```yaml
apiVersion: apps/v1
kind: Deployment
```

is valid.

But if the `kind` does not exist in the specified API version, Kubernetes will return an error.

---

## Examples of Kubernetes Kinds

Common `kind` values include:

```text
Pod
Service
Deployment
ReplicaSet
StatefulSet
DaemonSet
Job
CronJob
ConfigMap
Secret
Namespace
Ingress
PersistentVolumeClaim
```

---

# 3. metadata

The `metadata` field contains information used to identify and organize the object.

Common metadata fields include:

```yaml
metadata:
  name: nginx-pod
  namespace: default
  labels:
    app: nginx
  annotations:
    description: "Nginx test pod"
```

---

## metadata.name

The `name` field gives the object a unique name inside its namespace.

Example:

```yaml
metadata:
  name: nginx-pod
```

The name is used when running commands like:

```bash
kubectl get pod nginx-pod
kubectl describe pod nginx-pod
kubectl delete pod nginx-pod
```

---

## metadata.namespace

The `namespace` field defines where the object belongs.

Example:

```yaml
metadata:
  namespace: production
```

If no namespace is provided, Kubernetes usually uses the `default` namespace.

---

## metadata.labels

Labels are key-value pairs used to organize and select Kubernetes objects.

Example:

```yaml
metadata:
  labels:
    app: nginx
    environment: production
```

Labels are commonly used by:

* Services
* Deployments
* ReplicaSets
* selectors
* monitoring tools
* operators

Example Service selector:

```yaml
selector:
  app: nginx
```

This selector matches Pods with:

```yaml
labels:
  app: nginx
```

---

## metadata.annotations

Annotations are key-value pairs used for non-identifying information.

Example:

```yaml
metadata:
  annotations:
    description: "This pod runs nginx"
```

Annotations are different from labels.

Labels are usually used for selection and grouping.

Annotations are usually used for:

* extra metadata
* tool-specific configuration
* controller behavior
* documentation
* integration settings

Examples:

```yaml
metadata:
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
```

Annotations can affect object behavior when used by controllers or external tools.

---

## Kubernetes-Managed Metadata

Kubernetes may automatically add metadata to objects.

Examples:

* UID
* resource version
* creation timestamp
* managed fields
* last applied configuration annotation

This metadata helps Kubernetes manage and track resources internally.

---

# 4. spec

The `spec` field contains the actual desired configuration of the object.

Example:

```yaml
spec:
  containers:
    - name: nginx
      image: nginx
```

The shape of the `spec` depends on the object type.

A Pod `spec` looks different from a Deployment `spec`.

A Service `spec` looks different from an Ingress `spec`.

---

# Pod Manifest Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx
      ports:
        - containerPort: 80
```

For a Pod, the `spec` usually contains:

* containers
* volumes
* restart policy
* resource requests and limits
* environment variables
* probes
* security context

The `containers` field is required for most normal Pods.

---

# Deployment Manifest Example

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx
          ports:
            - containerPort: 80
```

For a Deployment, the `spec` contains different fields:

* replicas
* selector
* template
* strategy

The `template` section contains the Pod specification that the Deployment will create.

---

# Service Manifest Example

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: NodePort
  selector:
    app: nginx
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30080
```

For a Service, the `spec` usually contains:

* type
* selector
* ports

The selector determines which Pods the Service sends traffic to.

---

# Namespace Manifest Example

Some objects do not require a `spec`.

Example:

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: production
```

A Namespace is simple enough that it can be created without a `spec` section.

---

# Required Fields Depend on the Object Type

The required fields in `spec` depend on:

```text
apiVersion + kind
```

For example:

* a Pod requires container definitions
* a Deployment requires a selector and Pod template
* a Service requires ports and selectors
* an Ingress requires routing rules
* a Namespace may not need a spec

You do not need to memorize every field.

Instead, use:

* Kubernetes documentation
* IDE plugins
* schema validation
* `kubectl explain`
* `kubectl --dry-run=client -o yaml`

---

# Using kubectl explain

`kubectl explain` helps you inspect Kubernetes resource fields.

Examples:

```bash
kubectl explain pod
```

```bash
kubectl explain pod.spec
```

```bash
kubectl explain deployment.spec.template.spec.containers
```

This is useful when you are not sure which fields are available or required.

# Generating Manifests from Imperative Commands

Sometimes you may not remember the exact YAML structure for an object.

Kubernetes provides a useful way to generate YAML from an imperative command.

This is done using:

```bash
--dry-run=client -o yaml
```

---

# What Is --dry-run=client?

The `--dry-run=client` flag tells `kubectl`:

```text
Do not send this request to the Kubernetes API server.
Generate the object locally on the client side.
```

This means no object is created in the cluster.

The command only produces the YAML output.

---

# What Is -o yaml?

The `-o yaml` flag tells `kubectl`:

```text
Output the generated object as YAML.
```

Together:

```bash
--dry-run=client -o yaml
```

means:

```text
Generate the YAML manifest locally without creating the resource.
```

---

# Example: Generate a Pod Manifest

Suppose we want to create a Pod using the image:

```text
lm-academy/color-api:1.0.0
```

Imperative command:

```bash
kubectl run color-api-pod --image=lm-academy/color-api:1.0.0
```

To generate YAML instead of creating the Pod:

```bash
kubectl run color-api-pod \
  --image=lm-academy/color-api:1.0.0 \
  --dry-run=client \
  -o yaml
```

Example output:

```yaml
apiVersion: v1
kind: Pod
metadata:
  creationTimestamp: null
  labels:
    run: color-api-pod
  name: color-api-pod
spec:
  containers:
    - image: lm-academy/color-api:1.0.0
      name: color-api-pod
      resources: {}
  dnsPolicy: ClusterFirst
  restartPolicy: Always
status: {}
```

This output can be saved into a file and cleaned up if needed.

---

# Save Generated YAML to a File

You can redirect the generated YAML to a file:

```bash
kubectl run color-api-pod \
  --image=lm-academy/color-api:1.0.0 \
  --dry-run=client \
  -o yaml > color-api-pod.yaml
```

Then create the object:

```bash
kubectl create -f color-api-pod.yaml
```

or apply it declaratively:

```bash
kubectl apply -f color-api-pod.yaml
```

---

# Example: Generate a Service Manifest

Suppose a Pod named `nginx-pod` already exists.

To expose it as a NodePort Service:

```bash
kubectl expose pod nginx-pod \
  --type=NodePort \
  --port=80
```

To generate the Service YAML instead of creating it:

```bash
kubectl expose pod nginx-pod \
  --type=NodePort \
  --port=80 \
  --dry-run=client \
  -o yaml
```

Example output:

```yaml
apiVersion: v1
kind: Service
metadata:
  creationTimestamp: null
  labels:
    run: nginx-pod
  name: nginx-pod
spec:
  ports:
    - port: 80
      protocol: TCP
      targetPort: 80
  selector:
    run: nginx-pod
  type: NodePort
status:
  loadBalancer: {}
```

The important field here is:

```yaml
selector:
  run: nginx-pod
```

The Service uses this selector to find matching Pods.

---

# Why Generated YAML Is Useful

Generating YAML from commands is useful when:

* you are learning Kubernetes
* you forgot the manifest structure
* you want a quick starting point
* you want to avoid writing boilerplate manually
* you want to move from imperative commands to declarative YAML

However, generated YAML may contain unnecessary fields.

You should review and clean it before committing it to a repository.

```yaml

To create an object from a manifest:

```bash
kubectl create -f nginx-pod.yaml
```

This creates the object defined in the file.

If the object already exists, `kubectl create` usually fails.

```

# Applying Objects from Manifest Files

To declaratively create or update an object:

```bash
kubectl apply -f nginx-pod.yaml
```

If the object does not exist, it is created.

If the object already exists, Kubernetes tries to update it to match the manifest.

This is why `apply` is commonly used in production workflows.

# Migrating from create to apply

Sometimes you create a resource using:

```bash
kubectl create -f nginx-pod.yaml
```

Later, you may want to manage the same resource using the declarative `apply` command.

You do not always need to delete and recreate the object.

```

Kubernetes may show a warning about a missing annotation.

# Last Applied Configuration Annotation

When using `kubectl apply`, Kubernetes stores the last applied configuration in an annotation.

The annotation is:

```text
kubectl.kubernetes.io/last-applied-configuration
```

This annotation helps Kubernetes compare:

```text
previous applied configuration
current manifest
live cluster state
```

It is used to calculate changes during future `apply` operations.

---

# Missing Last Applied Annotation Warning

If a resource was created with `kubectl create`, it may not have the last applied configuration annotation.

When you later run:

```bash
kubectl apply -f nginx-pod.yaml
```

you may see a warning similar to:

```text
Warning: resource pods/nginx-pod is missing the kubectl.kubernetes.io/last-applied-configuration annotation
```

Kubernetes can usually patch this annotation automatically during the first `apply`.

After that, future `kubectl apply` commands work normally.

---

# Using --save-config with create

If you want `kubectl create` to store the last applied configuration from the beginning, use:

```bash
kubectl create -f nginx-pod.yaml --save-config
```

This saves the configuration annotation at creation time.

Then the object can be managed more cleanly with future `kubectl apply` commands.

---

# Checking the Last Applied Annotation

To view a resource as YAML:

```bash
kubectl get pod nginx-pod -o yaml
```

Look under:

```yaml
metadata:
  annotations:
    kubectl.kubernetes.io/last-applied-configuration: ...
```

If this annotation exists, the resource is ready for normal declarative management using `kubectl apply`.

# 15. Additional Important Concepts

---

## Init Containers

An init container runs before the main application containers start.

It is commonly used for:

* waiting for dependencies
* preparing configuration
* downloading startup assets
* running setup tasks

Important point:

```text
Init containers must complete successfully before normal containers start.
```

This is useful when the application should not start until some prerequisite work is finished.

---

## Headless Service

A headless Service is a Service without a cluster IP.

It is created using:

```yaml
spec:
  clusterIP: None
```

Headless Services are commonly used with StatefulSets.

They allow clients to discover the individual Pod DNS names instead of routing through a single virtual IP.

Useful for:

* databases
* distributed systems
* leader-follower applications
* peer-to-peer service discovery

---

## ResourceQuota

A ResourceQuota limits total resource usage inside a namespace.

It can restrict:

* total CPU
* total memory
* total storage
* total number of Pods
* total number of Services
* total number of PersistentVolumeClaims

This is useful in shared clusters to prevent one team or environment from consuming too many resources.

---

## LimitRange

A LimitRange defines default or minimum/maximum resource values for objects in a namespace.

It is often used together with ResourceQuota.

For example, a LimitRange can:

* set default CPU and memory requests
* set default CPU and memory limits
* enforce minimum resource values
* enforce maximum resource values

This helps teams avoid creating Pods with no resource settings at all.

---

## PodDisruptionBudget

A PodDisruptionBudget, or PDB, defines how many Pods must remain available during voluntary disruptions.

Voluntary disruptions include:

* node draining
* cluster upgrades
* maintenance operations

Example idea:

```text
At least 2 replicas must remain available
```

This is important for high availability during operational changes.

---

## Init Containers vs Sidecars

This is a useful interview comparison.

| Concept        | Init Container                            | Sidecar Container                          |
| -------------- | ----------------------------------------- | ------------------------------------------ |
| When it runs   | Before main containers                    | Alongside main containers                  |
| Main purpose   | Setup or preparation                      | Ongoing supporting behavior                |
| Lifecycle      | Ends before app starts                    | Runs with the Pod                          |
| Example use    | Wait for DB, render config, run migration | Logging proxy, metrics agent, mesh proxy   |

## OOMKilled

If a container exceeds its memory limit, Kubernetes may terminate it.

This often appears as:

```text
OOMKilled
```

Important idea:

* memory limits are enforced strongly
* CPU limits cause throttling rather than immediate termination

This is a very common troubleshooting and interview topic.

---

## CrashLoopBackOff

CrashLoopBackOff means a container keeps failing and Kubernetes keeps retrying it with backoff delays.

Common causes:

* bad startup command
* missing environment variables
* failed database connection
* application crash on boot
* invalid configuration

Common debugging steps:

```bash
kubectl describe pod <pod-name>
kubectl logs <pod-name>
kubectl logs <pod-name> --previous
```

---

## Eviction

Kubernetes may evict Pods when a node is under pressure.

Common pressure types:

* memory pressure
* disk pressure
* PID pressure

Eviction is different from an application crash.

It usually means the node itself is under resource stress.

## CNI

CNI stands for Container Network Interface.

It is the networking layer used to implement Pod networking in Kubernetes.

Examples:

* Calico
* Flannel
* Cilium
* Weave

CNI plugins are responsible for:

* Pod IP allocation
* network connectivity between Pods
* sometimes network policy enforcement

---

## CSI

CSI stands for Container Storage Interface.

It is the standard used by Kubernetes to integrate with storage systems.

CSI drivers allow Kubernetes to work with:

* cloud block storage
* network file systems
* external storage platforms

This is important for dynamic storage provisioning and persistent volumes.

## Gateway API

Ingress is still widely used, but modern Kubernetes networking increasingly also uses the Gateway API.

Gateway API aims to provide more expressive and role-oriented traffic management.

It is useful for:

* advanced routing
* shared platform networking
* clearer separation between infrastructure teams and application teams

For interviews, it is enough to know:

```text
Ingress is common and important.
Gateway API is a newer, more flexible traffic management model.
```

## 19.1 Labels and Selectors

Labels are key-value pairs attached to Kubernetes objects.

They help identify, organize, and group resources.

Common resources that use labels:

* Pods
* Nodes
* Services
* Deployments
* ReplicaSets

Labels are metadata, but unlike names, they are not unique.

That means multiple Pods can share the same label:

```yaml
app: color-api
```

This is normal and expected.

Labels are important because many Kubernetes objects rely on them for selection and association.

Examples:

* a Service uses labels to know which Pods should receive traffic
* a Deployment uses labels to know which Pods belong to it
* node selectors and affinity rules use labels to influence scheduling

### Why Labels Matter

Without labels, Kubernetes resources become much harder to organize.

Labels allow us to answer practical questions such as:

* Which Pods belong to this application?
* Which Pods are frontend versus backend?
* Which resources are in `dev` versus `prod`?
* Which release is `stable` versus `canary`?

Very common labels include:

* `app`
* `environment`
* `tier`
* `version`
* `release`

Example:

```yaml
metadata:
  labels:
    app: color-api
    environment: local
    tier: backend
    release: stable
```

### Selectors

Selectors are expressions used to filter objects based on labels.

Selectors are what make labels useful operationally.

If labels are the tags, selectors are the search conditions.

Two major selector types are used in Kubernetes:

* equality-based selectors
* set-based selectors

### Equality-Based Selectors

These are simple and very common.

They match exact key-value pairs.

Example:

```yaml
selector:
  matchLabels:
    app: color-api
    tier: backend
```

This matches Pods where both conditions are true:

* `app=color-api`
* `tier=backend`

This is effectively an AND condition.

### Set-Based Selectors

Set-based selectors are more expressive.

They are useful when matching based on a set of allowed or disallowed values.

Common operators:

* `In`
* `NotIn`
* `Exists`
* `DoesNotExist`

Example:

```yaml
selector:
  matchExpressions:
    - key: tier
      operator: In
      values:
        - frontend
        - backend
```

This matches any Pod whose `tier` is either `frontend` or `backend`.

Another example:

```yaml
selector:
  matchExpressions:
    - key: release
      operator: NotIn
      values:
        - canary
```

This matches Pods whose `release` is anything except `canary`.

This is useful when:

* you want to exclude one value
* there may be many allowed values
* you do not want to maintain a long positive allow-list

### `kubectl` Examples for Labels

Show label values in output:

```bash
kubectl get pods -L app
kubectl get pods -L app -L tier
```

Filter Pods by label:

```bash
kubectl get pods -l app=color-api
kubectl get pods -l tier=frontend
kubectl get pods -l 'app=color-api,tier=frontend'
kubectl get pods -l 'tier in (frontend,backend)'
kubectl get pods -l 'release notin (canary)'
```

### Labels and Deployments

Deployments rely heavily on labels.

A Deployment contains:

* a selector
* a Pod template with labels

Example:

```yaml
spec:
  selector:
    matchLabels:
      app: color-api
      tier: backend
  template:
    metadata:
      labels:
        app: color-api
        tier: backend
```

Important rule:

```text
The Deployment selector and the labels on the Pod template must align.
```

Otherwise the Deployment will not correctly manage the Pods it creates.

### Overlap Risk

A subtle but important operational issue is overlapping selectors.

Suppose:

* a standalone Pod has `app=color-api`
* a Deployment also selects `app=color-api`

Now `kubectl get pods -l app=color-api` will show both.

That does not mean the Deployment created both Pods.

It only means both match the label query.

This can become confusing very quickly.

One possible pattern is to add a distinguishing label such as:

```yaml
managed: deployment
```

Then selectors can be made more explicit.

Even so, in most real-world Deployments, simpler selectors are usually safer.

Complex selectors increase the risk of:

* accidental overlap
* difficult debugging
* unclear ownership between controllers

### Best Practices for Labels and Selectors

* Use a small, consistent label vocabulary.
* Keep Deployment selectors simple.
* Avoid overlapping selectors between different controllers.
* Use labels consistently across Pods, Services, and Deployments.
* Prefer clarity over cleverness.

## 19.2 Annotations

Annotations are also key-value pairs attached to Kubernetes objects.

They look similar to labels, but they serve a different purpose.

Main distinction:

* labels are for identifying and selecting
* annotations are for attaching non-identifying metadata

Annotations are commonly used for:

* tool-specific metadata
* controller configuration
* build information
* version information
* commit hashes
* runtime hints for operators

### Common Use Cases

Annotations often appear in:

* Ingress resources
* monitoring integrations
* logging agents
* custom controllers
* CI/CD metadata

Example conceptual annotation:

```yaml
metadata:
  annotations:
    nginx.ingress.kubernetes.io/force-ssl-redirect: "true"
```

This is not used for selection.

It is used by the Ingress controller to change behavior.

### Annotation Key Format

Annotations may include a DNS-style prefix:

```text
example.com/some-key
```

This prefix is especially common for:

* Kubernetes system components
* cloud providers
* third-party automation tools

If no prefix is used, the annotation is generally user-defined or private.

### Best Practices for Annotations

* Use labels for selection, not annotations.
* Use annotations when metadata should not affect grouping or matching.
* Prefer clear prefixes for system-generated or tool-managed annotations.

## 19.3 Namespaces

Namespaces are logical partitions inside a Kubernetes cluster.

They allow multiple groups of resources to coexist in the same physical cluster with logical separation.

A namespace can be used to separate:

* environments
* teams
* applications
* access boundaries
* resource quotas

Examples:

```text
dev
staging
prod
monitoring
```

### Why Namespaces Matter

Without namespaces, resources from unrelated applications or environments all live together in one shared space.

That increases risk.

For example:

* one team may accidentally impact another team
* one environment may consume too many resources
* access control becomes harder to define
* operational clarity becomes weaker

Namespaces create a meaningful unit of isolation inside a shared cluster.

### Built-In Namespaces

Kubernetes usually includes:

* `default`
* `kube-system`
* `kube-public`
* `kube-node-lease`

High-level meaning:

* `default`: resources go here when no namespace is specified
* `kube-system`: Kubernetes system components run here
* `kube-public`: publicly readable cluster information
* `kube-node-lease`: node lease tracking for health reporting

### Namespace YAML

Example:

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: dev
```

Namespaces do not typically have a `spec` section like many other resources.

### Working with Namespaces

List namespaces:

```bash
kubectl get namespaces
```

Create a resource in a namespace:

```yaml
metadata:
  name: color-api
  namespace: dev
```

Query resources inside a namespace:

```bash
kubectl get pods -n dev
kubectl describe pod color-api -n dev
```

### Setting a Default Namespace for `kubectl`

It can be convenient to set the current context namespace:

```bash
kubectl config set-context --current --namespace=dev
```

After this, commands such as `kubectl get pods` operate against `dev` by default.

This is useful, but also risky.

If you forget the active namespace, you may run commands against the wrong environment.

This matters especially for destructive commands such as:

```bash
kubectl delete deployment my-app
```

If you think you are in `dev` but you are actually in `prod`, the results can be severe.

### Inspecting Context

Useful commands:

```bash
kubectl config current-context
kubectl config view
```

These help confirm:

* which cluster you are pointing to
* which user is being used
* which namespace is active in the current context

### Listing Across All Namespaces

Sometimes you know the Pod name but not the namespace.

This is where:

```bash
kubectl get pods -A
```

becomes very useful.

### Deleting a Namespace

Deleting a namespace deletes the resources inside it.

That means:

* Pods
* Services
* Deployments
* ConfigMaps
* other namespaced resources

will be removed as part of namespace deletion.

This is why namespace deletion is a high-impact operation.

### Namespace Best Practices

* Use namespaces when there is a meaningful isolation boundary.
* Do not create namespaces for every tiny thing just because you can.
* Pair namespaces with RBAC.
* Pair namespaces with resource quotas in shared clusters.
* Use consistent naming such as `dev`, `staging`, and `prod`.

## 19.4 Service Communication Across Namespaces

Inside the same namespace, a Pod can usually reach a Service using only the Service name.

Example:

```text
color-api
```

But when a Pod in one namespace needs to talk to a Service in another namespace, the Service name alone is not enough.

In that case, use the fully qualified domain name:

```text
service-name.namespace.svc.cluster.local
```

Example:

```text
color-api.dev.svc.cluster.local
```

This tells cluster DNS exactly which Service to resolve.

### Why This Matters

Cross-namespace communication is common when:

* monitoring systems are centralized
* shared platform services exist
* one app depends on another service in a different namespace

### Practical Example

Suppose:

* `color-api` Service exists in namespace `dev`
* a traffic generator Pod exists in the `default` namespace

Then the traffic generator should target:

```text
color-api.dev.svc.cluster.local
```

instead of only:

```text
color-api
```

### Key Rule

```text
Same namespace: service name is often enough.
Different namespace: use the fully qualified service name.
```

## 19.5 Resource Quotas, Requests, and Limits

This is one of the most practical and operationally important areas in Kubernetes.

These concepts protect cluster stability and help ensure fair resource distribution.

### Resource Quotas

A ResourceQuota is a namespace-level policy.

It defines hard limits on what resources can be consumed inside that namespace.

Examples of what can be limited:

* CPU requests
* CPU limits
* memory requests
* memory limits
* storage
* object counts such as number of Pods

Example:

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: dev-quota
  namespace: dev
spec:
  hard:
    requests.cpu: "1"
    requests.memory: 1Gi
    limits.cpu: "2"
    limits.memory: 2Gi
```

This means all resources combined in the `dev` namespace must stay within those hard caps.

### Requests and Limits

Requests and limits are set at the container level.

They are not namespace-level values.

They describe what each container needs and what it is allowed to consume.

Requests:

* minimum resources used by the scheduler for placement
* guarantee-like scheduling baseline

Limits:

* upper bound of allowed resource usage

Example:

```yaml
resources:
  requests:
    cpu: 200m
    memory: 256Mi
  limits:
    cpu: 500m
    memory: 512Mi
```

If a Pod has multiple containers, each container may define its own requests and limits.

### CPU Units

Kubernetes often expresses CPU in millicores:

* `1000m` = 1 CPU
* `500m` = 0.5 CPU
* `200m` = 0.2 CPU

This is often easier to read for smaller values.

### Why Requests and Limits Matter

Without proper resource definitions:

* scheduling becomes less predictable
* noisy-neighbor issues become more likely
* quotas become harder to enforce correctly
* production behavior becomes less stable

### Quota and Scheduling Together

Think of quotas and requests/limits as two sides of the same system:

* the namespace quota defines the allowed budget
* the Pod resource settings define how much each Pod consumes from that budget

If the sum of all requested or limited resources exceeds the quota, new Pods may not be created.

### Practical Scheduling Example

Suppose the namespace quota is:

* `requests.cpu: 1`
* `requests.memory: 1Gi`
* `limits.cpu: 2`
* `limits.memory: 2Gi`

And one Pod uses:

* `requests.cpu: 200m`
* `requests.memory: 256Mi`
* `limits.cpu: 500m`
* `limits.memory: 512Mi`

Then four such Pods would consume:

* `800m` requested CPU
* `1Gi` requested memory
* `2` limited CPU
* `2Gi` limited memory

That fits exactly.

A fifth similar Pod would exceed quota and would not be allowed.

### Important Behavior

If quota is enforced, Kubernetes may reject Pods that do not specify resources appropriately.

In quota-controlled namespaces, requests and limits should be treated as standard practice.

### Inspecting Quota

Useful commands:

```bash
kubectl get resourcequota -A
kubectl describe resourcequota dev-quota -n dev
```

This shows:

* hard limits
* current used values

### What Happens When Quota Is Exceeded

When a Pod exceeds remaining quota:

* Pod creation may fail
* the API server can return a `forbidden` error
* messages often include requested, used, and limited values

This is a server-side enforcement mechanism.

### A Very Common Trap: Deployment Rollouts

A Deployment might fit perfectly under quota at steady state and still fail during rollout.

Why?

Because rolling updates temporarily create new Pods before old Pods are fully removed.

If the namespace is already near quota:

* old Pods still consume quota
* new Pods try to start
* total consumption exceeds quota
* rollout gets stuck

This is one of the easiest Kubernetes issues to underestimate.

### Diagnosing Quota-Blocked Rollouts

Useful commands:

```bash
kubectl rollout status deployment/color-api-deployment -n dev
kubectl get rs -n dev
kubectl describe rs <replicaset-name> -n dev
kubectl describe deployment color-api-deployment -n dev
```

The key detail is that the ReplicaSet is often where the clearest creation errors appear, because the ReplicaSet is the component attempting to create the Pods.

### Practical Ways to Fix Quota Issues

* Delete unused resources.
* Reduce requests and limits if they are oversized.
* Increase quota if infrastructure allows it.
* Adjust rollout strategy if temporary headroom is too small.
* Re-run or restart the rollout after fixing capacity.

### Quota Best Practices

* Apply quotas to shared namespaces.
* Size requests and limits based on realistic observations.
* Leave room for rollouts and scaling.
* Do not blindly over-reserve resources.
* Periodically review actual usage against declared values.

## 19.6 Health Probes: Startup, Liveness, and Readiness

Health probes are one of the biggest quality improvements you can make to Kubernetes workloads.

They allow Kubernetes to respond automatically to unhealthy or not-yet-ready containers.

### Why Probes Matter

A container can be in several imperfect states:

* process started, but app not fully initialized
* app stuck or deadlocked
* app running, but temporarily unable to serve traffic

From the outside, these can all look different operationally.

Kubernetes probes let us express those differences explicitly.

### Startup Probe

The startup probe checks whether the application started successfully.

If the startup probe fails repeatedly:

* Kubernetes restarts the container

This is especially useful for:

* slow-starting applications
* apps that need warm-up time
* avoiding premature liveness failures during startup

Example:

```yaml
startupProbe:
  httpGet:
    path: /up
    port: 80
  failureThreshold: 2
  periodSeconds: 3
```

Interpretation:

* probe `/up`
* try every 3 seconds
* if it fails enough times, restart the container

### Liveness Probe

The liveness probe checks whether the application is still healthy while running.

If it fails repeatedly:

* Kubernetes restarts the container

This helps recover from:

* deadlocks
* stuck event loops
* unhealthy but still-running processes

Example:

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 80
  failureThreshold: 3
  periodSeconds: 10
```

### Readiness Probe

The readiness probe checks whether the application is ready to receive traffic.

If it fails:

* the container is not restarted just because of readiness
* the Pod is removed from Service endpoints
* traffic is no longer routed to it

This is one of the most important behaviors in Kubernetes networking.

Example:

```yaml
readinessProbe:
  httpGet:
    path: /ready
    port: 80
  failureThreshold: 2
  periodSeconds: 5
```

### Behavioral Summary

Startup probe failure:

* container restart

Liveness probe failure:

* container restart

Readiness probe failure:

* no traffic sent to the Pod

### A Good Demo Pattern

The note material demonstrates a strong teaching pattern:

* one app image
* endpoints such as `/up`, `/health`, and `/ready`
* environment variables that intentionally cause failures

Examples:

* `DELAY_STARTUP`
* `FAIL_LIVENESS`
* `FAIL_READINESS`

This is a great way to simulate health scenarios without building many separate app versions.

### Startup Failure Scenario

If startup is artificially delayed longer than the startup probe tolerance:

* startup probe fails
* container restarts
* repeated failures may lead to `CrashLoopBackOff`

Useful commands:

```bash
kubectl get pod
kubectl describe pod <pod-name>
```

### Liveness Failure Scenario

If startup succeeds but `/health` later fails:

* container becomes unhealthy
* liveness probe fails
* Kubernetes restarts the container

This can repeat indefinitely until the underlying issue is fixed.

### Readiness Failure Scenario

This is where the traffic-handling behavior becomes most visible.

Suppose:

* a Deployment has 6 replicas
* some replicas fail readiness
* a Service fronts those Pods

Then:

* failed-readiness Pods keep running
* failed-readiness Pods do not receive Service traffic
* only ready Pods remain in the endpoint list

This is why readiness probes are so valuable for user-facing services.

### Verifying Readiness Behavior

Useful commands:

```bash
kubectl get pods
kubectl describe pod <pod-name>
kubectl describe svc <service-name>
kubectl logs <traffic-generator-pod> -f
kubectl get pods -o wide
```

If the readiness probe is doing its job, the Service endpoints will include only healthy Pod IPs.

### Probe Best Practices

* Always define readiness probes for traffic-serving apps.
* Define liveness probes only when you know what true unhealthy behavior looks like.
* Use startup probes for slow-starting workloads.
* Avoid overly aggressive thresholds that cause unnecessary restarts.
* Make probe endpoints lightweight and reliable.
